# Game Data — SkillSum Shared Engine

All game content lives in `packages/shared/src/`. Pure TypeScript, zero runtime deps — it runs in both the browser and the API server. **Never duplicate this logic in an app.**

## Levels (400 = 8 worlds × 50)

- Levels are **generated by factories**, not hand-written. `levels/builder.ts` holds the difficulty ramp (`slotFor`): X01-02 intro (no timer) → standard bands (90/75/60s) → X41-49 speedrun (45s) → bosses at every 10th, world boss at 50.
- Each world = one file `levels/worldN.ts` with a `WorldSpec` (operators, skillCycle, per-band number ranges, optional timerAdjust). Hand-tuned levels go in the `overrides` record keyed by local id (see `world1.ts`).
- Level ids: `w(worldId, localId) = (worldId-1)*50 + localId`. Never hardcode raw ids.
- Invariants enforced by `__tests__/levels.test.ts` — any change must keep these green:
  - ids exactly 1..400 unique; `worldId === ceil(id/50)`
  - `star1 < star2 < star3 <= targetScore`
  - every 10th level is a boss; intro levels untimed; timers ≥30s
  - `targetScore <= timeLimit / 2` (reachable at 2s/answer)
  - generator produces valid questions for every level's params

## Question generator (`generator.ts`)

One `switch` over `SkillType`. **NEVER `eval()` or `new Function()`** — this killed the old build. New skill type checklist:
1. Add to the `SkillType` union in `types.ts`
2. Add a switch case — answers must be non-negative integers; build division from `divisor × quotient`; guard subtraction non-negative
3. Add title + tip in `levels/builder.ts` (`SKILL_TITLES`, `SKILL_TIPS`)
4. Add fuzz test in `__tests__/generator.test.ts` (1000 iterations, seeded `mulberry32`)

## XP / stars (`xp.ts`)

Formula is spec'd in the master guide (`../SKILLSUM_2_0_MASTER_GUIDE.md`): base 10/correct, accuracy tiers (0.2–1.8×), speed bonus (<1500/2500/3500ms), streak bonus (5/10/20), mode bonuses (+100 complete/+200 boss/+150 PB), daily multiplier `min(1 + days×0.03, 2)`. Account level threshold `floor(500×(n-1)^1.5)`. Tuning changes must update `__tests__/xp.test.ts` table cases.

## Achievements (`achievements.ts`)

Exactly 50 (tested). Prefer declarative `condition` types over `custom`; `custom` needs a named `customId` handled in `checkAchievements`. The API stores unlocks in `user_achievements` — new achievements appear automatically, no migration.

## Daily challenges (`daily.ts`)

Deterministic per date: `hashDate` (FNV-1a) seeds `mulberry32`. Same date → same 3 tasks for all users. **No `Date.now()`/`Math.random()` in generation paths** — take the date/rng as input. Tests sample 365 dates.

## Gate

`npm test -w packages/shared && npm run typecheck` — all 57+ tests green before commit.
